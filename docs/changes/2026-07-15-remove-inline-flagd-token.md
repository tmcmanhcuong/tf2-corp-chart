# Change: Remove Inline flagd Sync Token

## Summary

Production flagd synchronization now obtains its bearer token from the `techx-corp-flagd-ui` Kubernetes Secret generated by External Secrets Operator (ESO), instead of storing the credential in `values-prod.yaml`. The chart passes the token through `FLAGD_SYNC_TOKEN`, keeps flagd on its existing singleton Critical Managed Node Group placement, and adds fail-closed render checks for the complete secret reference and command placeholder contract.

## Context

The production flagd HTTP source previously embedded its bearer credential in a Git-tracked values file. This exposed a reusable credential in repository history and coupled token rotation to source edits. The change is needed to move runtime credential delivery to the existing AWS Secrets Manager and ESO path without changing flagd scheduling or introducing direct mutation of Argo CD-managed resources.

Constraints shaping the implementation:

* Terraform must never read or store the token in variables, plans, state, outputs, or logs.
* AWS Secrets Manager token issuance and revocation are out-of-band operator actions, not implemented Git changes.
* The ESO-generated Secret must contain the replacement key before the application chart reference is merged.
* The existing flagd singleton Critical placement remains unchanged.

## Before

`values-prod.yaml` stored the flagd HTTP source bearer credential inline in the command JSON. The `<prefix>/flagd-ui` ESO mapping exposed only `SECRET_KEY_BASE`, and the Directive 03 verification script checked flagd placement and rollout behavior without checking its authentication source.

## After

`secrets-chart/templates/externalsecrets.yaml` maps both `SECRET_KEY_BASE` and `FLAGD_SYNC_TOKEN` from the existing `<prefix>/flagd-ui` AWS Secrets Manager object. `values-prod.yaml` injects `FLAGD_SYNC_TOKEN` from `techx-corp-flagd-ui` and sets the HTTP source header to `Bearer $(FLAGD_SYNC_TOKEN)`. The production verification script rejects any other source-level header value and checks the rendered Deployment for the exact Secret name, key, and placeholder command contract.

The operations guide requires a GitOps-ordered cutover: provision the replacement property out-of-band, sync and verify the ExternalSecret without reading payloads, merge the application reference, verify replacement pods, and revoke the superseded token last.

## Technical Design Decisions

* Reused the existing `flagd-ui` AWS Secrets Manager object and Kubernetes Secret rather than introducing a second object, keeping ownership and access policy aligned with the existing sidecar credential.
* Used component `envOverrides` with `secretKeyRef` because the chart already merges this structure into the flagd container environment.
* Used Kubernetes command environment expansion through `$(FLAGD_SYNC_TOKEN)` so the credential remains absent from rendered Git values while preserving the flagd source JSON contract.
* Added both source-level and rendered-manifest assertions. The source check prevents a literal or alternate header from entering `values-prod.yaml`; render checks prevent Secret wiring or template regressions.
* Kept the existing singleton Critical scheduling configuration unchanged because credential delivery does not alter the reviewed availability architecture.

## Implementation Details

1. Added `FLAGD_SYNC_TOKEN` to the documented ASM JSON shape and the flagd-ui ExternalSecret `data` mapping.
2. Added the flagd component environment override referencing `techx-corp-flagd-ui` key `FLAGD_SYNC_TOKEN`.
3. Replaced the production HTTP source header value with the runtime environment placeholder.
4. Extended `scripts/verify-directive-03.ps1` to fail closed on source and rendered-manifest deviations without printing credential content.
5. Updated the operations guide with the required GitOps ordering, key-name-only verification, Terraform exclusion, and safe rollback rules.

## Dependencies and Cross-Repository Impact

The existing AWS Secrets Manager object for each environment must receive a `FLAGD_SYNC_TOKEN` property before the secrets chart change is synced. This is an out-of-band secret operation and requires no Terraform code change; `techx-corp-infra` must not read or store the token. No `techx-corp-platform` change is required. The implementation changes only `techx-corp-chart`.

## Impact Analysis

| Dimension | Impact |
|---|---|
| **Application behavior** | flagd uses the same HTTP bearer authentication contract, with the token supplied at container startup from a Secret-backed environment variable. |
| **Infrastructure** | No new infrastructure resource; the existing ASM object and ESO-generated Secret gain one property/key. |
| **Deployment** | Requires ordered Argo CD reconciliation of the secrets chart before the application chart. |
| **Security** | Removes the active credential from the current Git configuration and keeps replacement values out of Terraform and render output. Repository history still requires incident-response handling. |
| **Reliability** | Deployment fails closed if the required Secret or key is absent; rollout must wait for the ExternalSecret to become Ready. |
| **Backward compatibility** | Compatible after the replacement ASM property exists; syncing the app reference first can cause `CreateContainerConfigError`. |
| **Observability** | Existing pod status and flagd logs remain the verification signals; secret payloads must not be logged. |

## Migration or Deployment Notes

1. Issue a replacement token and add it as `FLAGD_SYNC_TOKEN` to the existing environment-specific flagd-ui ASM object out-of-band. Never pass the value through Terraform.
2. Merge this secrets chart mapping first and allow Argo CD to reconcile it.
3. Verify ExternalSecret readiness and Kubernetes Secret key presence by key name only; do not retrieve payloads.
4. Merge the application chart reference and allow Argo CD to reconcile the flagd Deployment.
5. Verify the replacement pod and remote flag synchronization.
6. Revoke the superseded token only after successful verification.

## Risks and Rollback

| Risk | Likelihood | Severity | Mitigation / Rollback |
|---|---|---|---|
| Application sync occurs before the Secret key exists | Medium | High | Enforce secrets-first GitOps ordering; pause the app merge until key-name-only verification succeeds. |
| Replacement token is invalid or lacks required access | Low | High | Keep the superseded token valid until the new pod is healthy; issue another replacement out-of-band if verification fails. |
| Secret value appears in logs or terminal output | Low | High | Use key-name-only checks and never inspect environment variables or Secret payloads. |
| Repository history retains the previous credential | High | High | Revoke it after cutover and follow the repository incident-response process for history exposure. |

**Rollback procedure:**

Revert only to the last known-good secret-backed Git configuration and let Argo CD reconcile it. Keep ESO and the Kubernetes Secret in place. Never restore a literal, superseded, or revoked token to Git. If the replacement is unusable, issue a new replacement token out-of-band, update the existing ASM property, verify key presence without payload access, and repeat the ordered application reconciliation.
